## 블로그 본문 톤

- 두 대상을 비교하는 글(에디터, 도구, 라이브러리 등)에서는 "A가 B보다 낫다/부족하다"는 승패 프레이밍을 피한다.
- 약점은 "아직 부족하다", "이르다" 같은 폄하성 표현 대신, "이 영역은 기존에 검증된 방식(예: VSCode)을 그대로 쓴다"는 실용적·역할 분담 프레이밍으로 쓴다.
- 기능 지원 여부, 속도 차이 같은 객관적 사실은 그대로 서술한다 — 톤을 순화하는 것과 사실을 흐리는 것은 다르다.
- 결론에서 도구를 완전히 폐기하거나 "메인/서브"로 서열화하지 않는다. 작업 성격에 따라 병행한다는 식으로 마무리한다.
- 예시: [_posts/2026-04-30-zed-vscode.md](_posts/2026-04-30-zed-vscode.md) — Zed의 강점(속도, 협업)은 그대로 두고, Dev Container/Jupyter 미지원은 "VSCode의 통합을 그대로 활용한다"로 재구성한 사례.
- 전문용어를 처음 언급할 때는 풀네임과 공식 문서 링크를 문장에 자연스럽게 녹인다 (예: `[Language Server Protocol(LSP)](https://microsoft.github.io/language-server-protocol/)`). 목록 항목 안에 넣으면 다른 항목과 길이가 어긋나기 쉬우므로, 목록은 짧게 유지하고 설명은 목록 앞뒤의 별도 문장으로 뺀다.

커밋 메시지 형식은 AGENTS.md의 "커밋 메시지 형식" 절을 따른다 (위 임포트로 이미 포함됨).
